### Inconsistency in Data Sharing

![[Pasted image 20260502190702.png]]
![[Pasted image 20260502190709.png]]![[Pasted image 20260502191023.png]]

**Write-through** and **write-back** are two cache write policies used to maintain coherency between cache and main memory. In a **write-through cache**, every write updates both the cache and main memory immediately, ensuring memory always has the latest data; its advantages are simplicity and strong consistency, but it suffers from high memory traffic and lower performance. 
In contrast, a **write-back cache** updates only the cache and writes to main memory later (when the block is replaced), reducing memory traffic and improving performance; however, it is more complex, requires dirty bits, and can lead to temporary inconsistency since memory may hold stale data until updates occur.

### Process Migration and I/O

When a process migrates from one processor to another in a multiprocessor system, **cache inconsistency can occur** if the processors use private caches. Each processor may have its own cached copy of a shared variable (e.g., X), and these copies may not be updated consistently. This problem occurs in both **write-through** and **write-back** caches, although the cause differs slightly. Therefore, a **cache coherence protocol** is required to ensure correctness when processes migrate.

#### Two Protocol Approaches

In multiprocessor systems, maintaining cache coherence requires coordination between caches when shared data is accessed or modified. 
Two main approaches are used: **snoopy protocols** and **directory-based protocols**.

In bus-based systems, all processors are connected through a shared bus, which allows every cache to observe (or “snoop” on) all memory transactions. If a cache detects that another processor is modifying a memory location it holds, it can invalidate or update its local copy to maintain consistency; this forms the basis of snoopy protocols. However, in larger and more scalable systems that use point-to-point or multistage interconnection networks, such global observation is not feasible due to the lack of efficient broadcast mechanisms.

In these systems, **directory-based protocols** are used, where a centralized or distributed directory keeps track of which caches hold copies of each memory block and coordinates updates or invalidations. 

Overall, a cache coherence protocol defines the possible states of data in caches and memory, along with the rules and message exchanges needed to ensure all processors see a consistent view of shared data.

### The Futurebus+ Protocol

The **Futurebus+ protocol** is designed for **shared-memory multiprocessor systems**, supporting both **connected** and **split transactions** to improve scalability and performance. It provides flexible bus transactions that can support various cache coherence protocols, most notably the **MESI model**, which maintains consistency across multiple caches. The protocol extends traditional snoopy coherence mechanisms to work efficiently even in large systems with multiple interconnected buses.

### Protocol Performance Issues

The performance of snoopy cache coherence protocols depends strongly on **program behavior (workload patterns)** and **implementation efficiency**. The primary goal of snooping is to **reduce bus traffic**, while also improving **effective memory access time**. Several factors such as block size, type of protocol (write-invalidate vs write-update), and data sharing patterns significantly affect performance.

### Directory Structures

In multistage network-based multiprocessors, cache coherence cannot rely on broadcast mechanisms as in bus-based systems. Instead, **directory-based protocols** are used, where a **cache directory keeps track of which caches hold copies of each memory block**. The directory stores information about the **state of the block** and the **locations (presence) of its copies**, enabling targeted communication rather than system-wide broadcasting. Different directory schemes vary in how this information is stored and managed.

In a **full-map directory**, each memory block has **one bit per processor**, so the directory entry size grows with the number of processors (**O(N)**). Since this entry exists for every memory block (and total memory itself scales with system size, also **O(N)**), the **total memory overhead becomes O(N²)**. This quadratic growth makes full-map directories **impractical for large systems**.
### Limited Directories

Limited directory schemes are designed to **reduce directory size** by restricting the number of cache copies tracked per memory block. Instead of storing information for all processors, each directory entry maintains only a **fixed number of pointers (i < N)**.

Scale as $O(N\log_2 N)$

### Chained Directories

![[Pasted image 20260502224407.png]]
![[Pasted image 20260502224354.png]]

To handle this, two approaches are possible. In the first approach, the system updates the chain by sending a message to neighboring caches so that Ci is removed and the chain is reconnected (splicing). This maintains correctness but is complex because it requires traversing and updating pointers in the chain. In the second approach, the system simply invalidates the copy of X in Ci via another cache, avoiding detailed pointer updates. This method is simpler to implement but may involve additional invalidations.

In both cases, sequential consistency must be preserved, which is achieved by temporarily locking the memory location while invalidation or updates are performed. An alternative improvement is to use a doubly linked chain, where each cache maintains both forward and backward pointers. This allows efficient removal of a cache from the chain without traversal, but it increases memory overhead, message size, and protocol complexity.

Despite these challenges, chained-directory protocols remain scalable because they use a fixed number of pointers per block, and each pointer only requires log₂N bits. This means their memory overhead grows slowly (logarithmically) with the number of processors, making them suitable for large systems compared to full-map directories.

Scale as $O(N)$
