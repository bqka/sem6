### Multiple Functional Units

![[Pasted image 20260502004606.png]]

In a pipelined processor, a bottleneck often occurs in the **execute stage** because many instructions may require execution at the same time, but a single functional unit cannot handle them all. This limits throughput even if other stages are free. To overcome this, modern processors use **multiple functional units (FUs)** such as ALUs, floating-point units, and memory units. This allows several instructions to execute **in parallel**, significantly improving performance.

However, parallel execution introduces the problem of **data dependencies**. For example, if one instruction computes `R1 = R2 + R3` and the next instruction needs `R1` (`R4 = R1 × R5`), the second instruction cannot proceed until the first one produces its result. Instead of stalling the entire pipeline, the processor uses **reservation stations (RS)**. These are small buffers placed before each functional unit that hold instructions until their operands are ready. For instance, the multiply instruction can wait in a reservation station until `R1` becomes available, while other independent instructions continue executing.

```
Instruction arrives → goes to RS
If data ready → execute immediately
If not → wait in RS
```

To manage dependencies efficiently, the system uses a **tagging mechanism**. Each result is assigned a tag identifying the functional unit that will produce it. If an operand is not yet available, the reservation station stores the tag instead of the value. When the result is eventually produced, it is broadcast along with its tag, and any waiting instruction that has the same tag captures the value. For example, if `R1` is being computed by one unit, dependent instructions track its tag and automatically receive the result when ready, without needing to stall.

```
R1 not ready → tagged as coming from FU1
```


This combination of **multiple functional units, reservation stations, and tagging** allows the processor to execute instructions **out of order** while still preserving correct results. Independent instructions such as a load (`R6 = Mem(A)`) or another arithmetic operation can proceed even if earlier instructions are waiting for data. This removes the execution-stage bottleneck, improves resource utilization, and increases overall pipeline throughput.

```
Fetch → Decode → Issue → Reservation Stations → Functional Units → Writeback
```

### Hazard Avoidance

![[Pasted image 20260502010251.png]]
![[Pasted image 20260502010405.png]]

![[Pasted image 20260502010450.png]]

A **Read After Write (RAW) hazard** occurs when an instruction needs to read a value that a previous instruction has not yet written. This corresponds to **flow dependence**. For example, if instruction I computes `R1 = R2 + R3` and the next instruction J uses `R4 = R1 × R5`, then J must wait until I writes the correct value of R1. If J reads R1 too early, it gets an incorrect value, causing a hazard.

A **Write After Write (WAW) hazard** occurs when two instructions write to the same destination. This is called **output dependence**. For example, if instruction I writes `R1 = R2 + R3` and a later instruction J writes `R1 = R4 + R5`, the final value of R1 depends on the correct execution order. If J writes before I finishes, the result of I may overwrite J’s result incorrectly.

A **Write After Read (WAR) hazard** occurs when an instruction writes to a location that a previous instruction still needs to read. This is known as **antidependence**. For example, if instruction I reads `R1` in `R2 = R1 + R3`, and instruction J later writes `R1 = R4 + R5`, then J must not write to R1 before I has completed reading it. Otherwise, I will read the wrong value.

These conditions (RAW, WAR, WAW) indicate when hazards are **possible**, but they are not always guaranteed to occur. Whether a hazard actually happens depends on the **timing and execution order** of instructions in the pipeline. If instructions are executed in a safe order or if results are available in time (e.g., via forwarding), the hazard can be avoided.

To handle hazards, processors use **hardware mechanisms such as tagging**. Each register may have a tag bit indicating whether its value is ready or still being produced. When instructions are loaded into buffers (like prefetch or reservation stations), these tags help track dependencies. As instructions read or write registers, the tag bits are updated, ensuring that operations occur in a safe order and hazards are avoided without unnecessarily stalling the pipeline.

Dynamic branch prediction improves performance by using the **past behavior of branch instructions during program execution**. To do this, the processor must maintain **history information**, which requires additional hardware such as storage bits and update logic. However, storing too much history increases hardware complexity and cost, so in practice only a **small amount of recent history** (e.g., 1-bit or 2-bit) is used.

## Superscalar Pipeline Design

![[Pasted image 20260502160725.png]]

A **superscalar processor** is designed to issue multiple instructions per cycle. If the processor has degree _m_, it can issue up to _m_ instructions simultaneously. To fully utilize such a processor, the program must provide sufficient **instruction-level parallelism (ILP)** equal to _m_. A scalar processor is simply a special case where _m = 1_.
